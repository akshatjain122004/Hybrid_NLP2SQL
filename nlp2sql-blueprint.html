Architecture Deep-Dive
All 9 Layers
Every layer is independently testable. Build and validate each one before connecting to the next. The goal is that each layer should reduce the load on the one after it.
L1 · LLM-Free

Pre-Processing Layer

First touch on raw user input — clean, normalize, tokenize before anything else runs

Normalizer
pipeline/normalizer.py
Lowercases input, strips filler phrases ("can you please", "tell me", "I want to know"), corrects common spelling errors using pyspellchecker, and standardizes date formats ("last month", "Q3", "yesterday"). This alone eliminates a massive amount of variation.
pyspellchecker re (regex) python-dateutil
IN →Raw user string
OUT →Cleaned, lowercase string
Tokenizer + POS Tagger
pipeline/tokenizer.py
Tokenizes and runs Part-of-Speech tagging via spaCy. Identifies nouns (likely entity values or table names), verbs (action: show/find/count), adjectives (filters: highest/latest), and numbers. These tags feed into intent classification and entity extraction downstream.
spaCy (en_core_web_sm)
IN →Cleaned string
OUT →Token list with POS tags
L2 · LLM-Free

Understanding Layer

Classifies what the user wants, extracts values, and links query words to your actual schema

Intent Classifier
pipeline/intent_classifier.py
A lightweight sklearn classifier (Logistic Regression or SVM) trained on 200–300 labeled (query, intent) examples. Classifies queries into: lookup, filter, aggregation, top_n, comparison, group_by. Also outputs a confidence score — the key signal for routing later. This model is fast (<5ms) and needs no API call.
scikit-learn TF-IDF vectorizer joblib (persist)
IN →Cleaned tokens + POS tags
OUT →intent: str, confidence: float
Entity Extractor
pipeline/entity_extractor.py
Extracts concrete values from the query: dates/ranges, numeric thresholds, product names, category names, user IDs. Uses spaCy NER for named entities and custom regex patterns for domain-specific values. Output is a typed dict of extracted slot values to fill into the IR.
spaCy NER regex python-dateutil
IN →Tokens + POS
OUT →{"date_range": ..., "limit": 10, ...}
Schema Linker
pipeline/schema_linker.py
The most critical component. Maps words from the user query to your actual table and column names. Uses fuzzy string matching to handle typos and synonyms ("staff" → employees, "revenue" → total_amount). Backed by a manually curated synonyms.json dictionary for your e-commerce domain. Reports a link confidence score per token.
rapidfuzz synonyms.json schema.json
IN →Entities + all query tokens
OUT →{"products": [...cols], "orders": [...cols]}
L3 · LLM-Free

Semantic Cache Layer

Checks Qdrant for near-duplicate past queries — cache hit = skip layers 4 & 5 entirely

Semantic Cache
cache/semantic_cache.py
Encodes the normalized query using sentence-transformers and searches Qdrant for the nearest past query above a cosine similarity threshold (e.g., 0.92). On a hit, retrieves the cached SQL template and swaps in the newly extracted entity values. On a miss, passes through to the Resolution layer. This eliminates LLM calls for all repeat-pattern queries — in real e-commerce usage, 40–60% of queries follow repeating patterns.
Qdrant sentence-transformers all-MiniLM-L6-v2
HIT →SQL template with slot-filled values → skip to L7
MISS →Continue to L4
Cache Warmer
cache/cache_warmer.py
A background service (run as a cron or startup task) that seeds Qdrant with nl_sql_pairs.json — your manually curated list of known (NL query, SQL) pairs. Also runs nightly to embed and load any new confirmed pairs from the logger. This is how your cache improves over time without touching the model.
Qdrant client APScheduler
IN →nl_sql_pairs.json / logger DB
OUT →Qdrant collection updated
L4 · LLM-Free

Resolution Layer

Builds the Intermediate Representation (IR) — a structured object that compiles to SQL without any ML

IR Builder
pipeline/ir_builder.py
Constructs the Intermediate Representation — a Python dataclass or typed dict with fields: select, from_, joins, where, group_by, order_by, limit. Fills each field from the outputs of L2: intent → select/aggregation shape, entities → where conditions, schema links → table/column names. The IR is language-agnostic and maps cleanly to SQL clauses.
Python dataclass
IN →intent, entities, schema links
OUT →IR dataclass (structured)
JOIN Resolver
db/schema_graph.py
Models your e-commerce database schema as a graph (networkx). Nodes are tables; edges are foreign-key relationships. When the IR requires data from two or more tables, BFS finds the shortest path between them and adds the correct JOIN chain automatically. This replaces one of the hardest things LLMs do in NL2SQL — JOIN inference — with a deterministic graph traversal.
networkx schema.json
IN →Required tables from IR
OUT →JOIN chain added to IR
Template Engine
pipeline/template_engine.py
For the most common query patterns (top-N products, total revenue by period, orders per customer, etc.), stores pre-written SQL templates as JSON with named slots. Matches the current intent + entity combination to a template fingerprint. On match, fills slots with extracted values — zero SQL generation needed. Run this before IR Builder for the fastest possible path.
templates.json Jinja2 (optional)
HIT →Fully filled SQL string → skip to L7
MISS →Fall through to IR Builder
L5 · LLM-Free

SQL Compilation Layer

Deterministically converts the IR into a valid SQL string — pure Python, no model, no ambiguity

IR → SQL Compiler
pipeline/sql_compiler.py
A pure Python function that reads the IR dataclass and serializes it into a valid SQL string. Maps: select → SELECT clause, from_ + joins → FROM/JOIN, where → WHERE (with proper AND chaining), group_by → GROUP BY, order_by → ORDER BY with direction, limit → LIMIT. Because the IR is already structured and validated, this function never fails if the IR is valid. It's just string assembly.
Pure Python No dependencies
IN →IR dataclass (complete)
OUT →Raw SQL string
L6 · LLM Decision Gate

Confidence Scoring + LangGraph Routing

Scores how certain the pipeline is, then routes: deterministic path or LLM fallback

Score pipeline output
→
Composite confidence score
→
Score ≥ 0.75 → Compiled SQL
or
Score < 0.75 → LLM Fallback
or
Unsafe → Reject
Threshold is configurable. Start at 0.75, tune based on observed error rates.
Confidence Scorer
router/confidence_scorer.py
Combines 4 signals into a single 0–1 confidence score: (1) intent classifier probability, (2) schema linker match score (rapidfuzz ratio), (3) entity extraction completeness (all expected slots filled?), (4) IR completeness (no null required fields). Weighted average — you tune weights based on which signals are most reliable in your domain.
Pure Python Weighted average
IN →Intent conf, schema score, entity fill rate
OUT →confidence: float (0.0–1.0)
LangGraph Router
router/langgraph_router.py
A LangGraph StateGraph with three conditional edges: (1) HIGH confidence → pass compiled SQL to validation, (2) LOW confidence → invoke LLM fallback node, (3) UNSAFE → reject immediately. State carries the full pipeline context so the LLM fallback receives pre-linked schema, extracted entities, and detected intent — not just the raw query.
LangGraph StateGraph
IN →Pipeline state + confidence score
OUT →Route decision + SQL or LLM call
LLM Fallback Handler
llm/fallback_handler.py
Invoked only on low-confidence queries. Builds a narrow prompt: pre-linked table names, only the relevant column subset (not the full schema), detected intent, and extracted entities. This makes the LLM call much cheaper and more accurate than a raw NL→SQL prompt. Tags the result as LLM-generated for the miss tracker.
OpenAI / Gemini Narrow prompt
IN →Linked schema subset + context
OUT →SQL string + source="llm" tag
L7 · Always Runs

Validation Layer

Every SQL — whether compiled or LLM-generated — passes through this before touching the database

SQL Syntax Validator
validator/syntax_validator.py
Parses the SQL string with sqlglot before execution. Catches syntax errors, unknown functions, and malformed clauses. sqlglot supports multiple SQL dialects — configure for your target DB (PostgreSQL). If parsing fails, the query is rejected immediately with a structured error, never sent to the database.
sqlglot
Read-Only Enforcer
validator/safety_guard.py
Operates at two levels: (1) string-level: rejects any query containing INSERT, UPDATE, DELETE, DROP, TRUNCATE, ALTER, CREATE, GRANT — even in comments. (2) AST-level: uses sqlglot's parsed AST to confirm the root node is a SELECT statement. Double-layer protection so no prompt injection can bypass it.
sqlglot AST regex
Table/Column Whitelist
validator/whitelist.py
Extracts all table and column references from the parsed AST and checks each one against a whitelist loaded from schema.json. Rejects queries accessing tables or columns outside the permitted list. Prevents data exfiltration and schema enumeration attacks, especially important when the LLM fallback is active.
sqlglot AST schema.json
Dry Run (EXPLAIN)
validator/dry_run.py
Sends the validated query to the DB as EXPLAIN (not EXPLAIN ANALYZE) — this computes the query plan without executing it. Catches missing-table errors, type mismatches, and performance-catastrophic queries (e.g., cross-join with no predicate). Gate: reject if EXPLAIN returns an error or estimated rows > configurable threshold.
psycopg2 / SQLAlchemy EXPLAIN
L8 · LLM-Free

Execution Layer

Runs the validated SQL, formats results into structured JSON for the API response

Query Executor
db/executor.py
Executes the SQL using a connection pool (SQLAlchemy). Enforces a query timeout (configurable — default 10s). Handles empty result sets gracefully. Returns raw rows as list of dicts using cursor.description for column names. This is the only point of contact with the actual database — everything before this is stateless processing.
SQLAlchemy Connection pool
IN →Validated SQL string
OUT →List[Dict] raw results
Result Formatter
db/result_formatter.py
Converts raw query results into a clean API response: JSON with columns, rows, row_count, execution_time_ms, source (deterministic | template | cache | llm), and the generated SQL itself (for debugging/transparency). The source field is critical — it tells you and your users how the answer was derived.
Pure Python
IN →Raw rows, metadata
OUT →Structured JSON response
L9 · Feedback Loop

Feedback Layer

Every query teaches the system — cache expands, LLM bypass rate improves over time

Query Logger
feedback/query_logger.py
Logs every query tuple: (timestamp, nl_query, generated_sql, source, execution_time_ms, row_count, success). Stored in a lightweight PostgreSQL table or SQLite for local dev. This log is your training data factory — every confirmed success is a candidate for a new cache entry or template.
SQLAlchemy PostgreSQL
Miss Tracker
feedback/miss_tracker.py
Flags all queries where source="llm" and queues them for review. After manual verification that the LLM-generated SQL was correct, these pairs are promoted to nl_sql_pairs.json for cache seeding. This is how you systematically shrink the LLM fallback rate: each verified miss becomes a future cache hit.
Query logger DB
Goal →LLM fallback rate falls as cache grows
Intent Dataset Grower
feedback/dataset_grower.py
Extracts new query + intent pairs from the log (using the intent the classifier assigned, confirmed by successful execution). Appends to intent_dataset.csv. Periodically retrain the sklearn intent classifier on the expanded dataset to improve low-confidence cases. The classifier gets better as traffic grows.
pandas scikit-learn
Goal →Classifier accuracy improves over time
Worked Example
Query → IR → SQL
How the pipeline processes: "show me the top 10 products by revenue in the last month"
🟢 Full deterministic path — no LLM involved
L1

After Pre-Processing

"show me the top 10 products by revenue in the last month"
→ "top 10 products revenue last month"

L2

After Understanding

Intent: top_n (conf: 0.93)  |  Entities: limit=10, date_range=last_month
Schema links: "products" → products table, "revenue" → order_items.subtotal

L3

Cache Check

Similar query found with score 0.88 — below 0.92 threshold (different date range). Cache miss → continue.

L4

IR Built

Intermediate Representation (IR)
{
  "intent": "top_n",
  "select": [
    "p.product_name",
    "SUM(oi.subtotal) AS total_revenue"
  ],
  "from_": "products p",
  "joins": [
    {
      "table": "order_items oi",
      "on": "p.id = oi.product_id"
    },
    {
      "table": "orders o",
      "on": "oi.order_id = o.id"
    }
  ],
  "where": [
    "o.created_at >= DATE_TRUNC('month', NOW() - INTERVAL '1 month')",
    "o.created_at < DATE_TRUNC('month', NOW())"
  ],
  "group_by": ["p.product_name"],
  "order_by": [
    {"col": "total_revenue", "dir": "DESC"}
  ],
  "limit": 10
}
Compiled SQL (from ir_compiler.py)
SELECT
  p.product_name,
  SUM(oi.subtotal) AS total_revenue
FROM products p
JOIN order_items oi
  ON p.id = oi.product_id
JOIN orders o
  ON oi.order_id = o.id
WHERE
  o.created_at >= DATE_TRUNC(
    'month', NOW() - INTERVAL '1 month'
  )
  AND o.created_at < DATE_TRUNC(
    'month', NOW()
  )
GROUP BY p.product_name
ORDER BY total_revenue DESC
LIMIT 10;

↑ Zero LLM calls. JOINs resolved via BFS on the schema graph. Date range from entity extractor. Pure Python compilation.

Development Roadmap
Build Sequence — 11 Phases
Follow this exact order. Each phase produces a testable output before moving on. Never skip ahead — the IR and schema linker are load-bearing; everything plugs into them.

  • Set up virtual env, install all dependencies, write requirements.txt
  • Write schema.json — your complete e-commerce schema: tables, columns, types, FK edges
  • Write synonyms.json — first pass: 20–30 domain-specific word→column mappings
  • Docker Compose: FastAPI + Qdrant + PostgreSQL running locally
schema.json, synonyms.json, docker-compose.yml — project boots cleanly

Build

normalizer + tokenizer tested on 50 queries — clean token lists with POS
Phase 2
Schema Linker — The Critical One

Non-Negotiables
Rules to Build By
Build schema.json first
Every other component is driven by it. If your schema changes, update schema.json + synonyms.json — not the code.
Test each layer in isolation
Feed hand-crafted inputs to each component before wiring them together. A bug in the schema linker will silently corrupt everything downstream.
Log everything from day one
Your log is your training data. Every query that reaches the LLM is an opportunity — if verified correct, it becomes a future cache entry.
Never trust LLM SQL directly
